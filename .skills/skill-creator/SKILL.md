# Skill Creator

新しいAgent Skillを作成するためのスキル。
Agent Skills仕様（ https://agentskills.io/specification ）に準拠したスキルを生成する。

## 手順

### 1. 要件の確認

ユーザーに以下を確認する：

- **スキルの目的**: 何をするスキルか
- **スキル名**: kebab-case（小文字英数字とハイフン）で命名する
- **使用場面**: どのような状況・プロンプトでこのスキルが呼び出されるべきか

不明な点があれば、先に質問して明確にすること。推測で進めない。

### 2. スキルの作成

以下の構成でファイルを作成する：

```
.skills/<スキル名>/
└── SKILL.md
```

必要に応じて補助ファイルを追加する：

```
.skills/<スキル名>/
├── SKILL.md
├── scripts/       # 実行可能なスクリプト
├── references/    # 補足ドキュメント
└── assets/        # テンプレートやデータファイル
```

補助ファイルを追加すべきサイン：
- 複数の場面で同じ補助コードを毎回生成することになりそうな場合 → `scripts/` にバンドルする
- 本文が長くなりすぎる場合（目安: 300行超） → 詳細を `references/` に分離する

### 3. SKILL.md の記述

#### フロントマター（必須）

```yaml
---
name: スキル名
description: スキルの説明。何をするか、いつ使うかを明記する。
metadata:
  version: "0.1"
---
```

フロントマターのルール：
- `name`: 必須。1〜64文字。小文字英数字とハイフンのみ。ハイフンで始終しない。連続ハイフン不可。親ディレクトリ名と一致させる。
- `description`: 必須。1〜1024文字。書き方は後述の「description の書き方」を参照。
- `metadata`: 任意。versionの記載を推奨。

その他のオプションフィールド（`license`、`compatibility`、`allowed-tools`）については [Agent Skills仕様](https://agentskills.io/specification) を参照。

#### description の書き方

descriptionはスキルが呼び出されるかどうかを決める最重要フィールドである。
AIエージェントはdescriptionを見てスキルを使うかどうか判断するため、以下の点に注意して書く。

**「何をするか」と「いつ使うか」の両方を含める。** この2つの情報をdescriptionに集約する。本文に分散させない。

**やや積極的に書く。** AIエージェントはスキルの使用に慎重になる傾向がある。ユーザーが正確なキーワードを言わなくてもスキルが発動するよう、関連する状況を幅広く記述する。

悪い例：
```yaml
description: ダッシュボードを作成する。
```

良い例：
```yaml
description: ダッシュボードを作成する。データの可視化、メトリクスの表示、グラフやチャートの作成に関する依頼があった場合にも使用する。ユーザーが「ダッシュボード」と明言しなくても、データを視覚的に表示したい意図があれば使用する。
```

#### 本文

フロントマターの後に、Markdown形式でスキルの指示を記述する。

**含めるべき内容：**
- スキルの概要（1〜2文）
- 手順（ステップバイステップ）
- 具体的な入出力の例
- エッジケースや注意事項
- 出力形式がある場合はテンプレートを明示する

**本文を書く際の原則：**

1. **「なぜそうするか」を説明する。** 指示だけでなく、その背景にある意図を書く。AIは意図を理解すれば、想定外の状況にも適切に対応できる。
   - 悪い例: 「見出しは必ず##で始めること」
   - 良い例: 「見出しは##で始める。#はドキュメントタイトル用に予約し、階層構造を明確にするため」

2. **命令形で簡潔に書く。** 指示文は命令形を使い、冗長な表現を避ける。

3. **過度に硬い制約を避ける。** 「必ず」「絶対に」を多用するのは危険信号。AIが状況に応じて適切に判断できるよう、背景情報を与える方が効果的。

4. **無駄な指示を省く。** スキルの指示がAIに非生産的な作業をさせていないか確認する。不要な手順は削除する。

5. **特定のAIエージェントに依存した記述を避ける。** 特定ツールの関数名やエージェント固有のコマンドを直接記述しない。

6. **段階的情報開示を意識する。** スキルの情報は3階層で読み込まれる：
   - メタデータ（name + description）: 常に読み込まれる（約100トークン）
   - SKILL.md本文: スキルが選択されたとき（5000トークン以下推奨）
   - 補助ファイル（scripts/references/assets）: 必要なときだけ

   本文には「次に何を読むべきか」のポインタを明示する。

7. **500行以内に収める。** 超えそうな場合は `references/` に分離し、SKILL.mdから相対パスでリンクする。参照ファイルが300行を超える場合は目次を付ける。ファイル参照は1階層の深さまでに留め、深いネストを避ける。

8. **少数の例に過度に最適化しない。** スキルは多様なプロンプトに対応する必要がある。特定の例だけにうまく機能する書き方ではなく、一般的なパターンとして機能する書き方を選ぶ。

### 4. 作成後の確認

作成したスキルについて、以下を確認する：

- [ ] `name` がディレクトリ名と一致しているか
- [ ] `description` が機能と使用場面の両方を説明しているか。やや積極的に書かれているか
- [ ] 本文の手順が明確で、AIエージェントが従えるか
- [ ] 指示に「なぜそうするか」の説明があるか
- [ ] 過度な制約（「必ず」「絶対に」の多用）がないか
- [ ] 特定のAIエージェントに依存していないか
- [ ] 繰り返し生成されそうな補助コードが `scripts/` にバンドルされているか
